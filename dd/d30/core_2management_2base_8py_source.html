72 class BaseCommand(object):
73  """
74  The base class from which all management commands ultimately
75  derive.
76 
77  Use this class if you want access to all of the mechanisms which
78  parse the command-line arguments and work out what code to call in
79  response; if you don't need to change any of that behavior,
80  consider using one of the subclasses defined in this file.
81 
82  If you are interested in overriding/customizing various aspects of
83  the command-parsing and -execution behavior, the normal flow works
84  as follows:
85 
86  1. ``django-admin.py`` or ``manage.py`` loads the command class
87  and calls its ``run_from_argv()`` method.
88 
89  2. The ``run_from_argv()`` method calls ``create_parser()`` to get
90  an ``OptionParser`` for the arguments, parses them, performs
91  any environment changes requested by options like
92  ``pythonpath``, and then calls the ``execute()`` method,
93  passing the parsed arguments.
94 
95  3. The ``execute()`` method attempts to carry out the command by
96  calling the ``handle()`` method with the parsed arguments; any
97  output produced by ``handle()`` will be printed to standard
98  output and, if the command is intended to produce a block of
99  SQL statements, will be wrapped in ``BEGIN`` and ``COMMIT``.
100 
101  4. If ``handle()`` or ``execute()`` raised any exception (e.g.
102  ``CommandError``), ``run_from_argv()`` will instead print an error
103  message to ``stderr``.
104 
105  Thus, the ``handle()`` method is typically the starting point for
106  subclasses; many built-in commands and command types either place
107  all of their logic in ``handle()``, or perform some additional
108  parsing work in ``handle()`` and then delegate from it to more
109  specialized methods as needed.
110 
111  Several attributes affect behavior at various steps along the way:
112 
113  ``args``
114  A string listing the arguments accepted by the command,
115  suitable for use in help messages; e.g., a command which takes
116  a list of application names might set this to '<appname
117  appname ...>'.
118 
119  ``can_import_settings``
120  A boolean indicating whether the command needs to be able to
121  import Django settings; if ``True``, ``execute()`` will verify
122  that this is possible before proceeding. Default value is
123  ``True``.
124 
125  ``help``
126  A short description of the command, which will be printed in
127  help messages.
128 
129  ``option_list``
130  This is the list of ``optparse`` options which will be fed
131  into the command's ``OptionParser`` for parsing arguments.
132 
133  ``output_transaction``
134  A boolean indicating whether the command outputs SQL
135  statements; if ``True``, the output will automatically be
136  wrapped with ``BEGIN;`` and ``COMMIT;``. Default value is
137  ``False``.
138 
139  ``requires_model_validation``
140  A boolean; if ``True``, validation of installed models will be
141  performed prior to executing the command. Default value is
142  ``True``. To validate an individual application's models
143  rather than all applications' models, call
144  ``self.validate(app)`` from ``handle()``, where ``app`` is the
145  application's Python module.
146 
147  ``leave_locale_alone``
148  A boolean indicating whether the locale set in settings should be
149  preserved during the execution of the command instead of being
150  forcibly set to 'en-us'.
151 
152  Default value is ``False``.
153 
154  Make sure you know what you are doing if you decide to change the value
155  of this option in your custom command if it creates database content
156  that is locale-sensitive and such content shouldn't contain any
157  translations (like it happens e.g. with django.contrim.auth
158  permissions) as making the locale differ from the de facto default
159  'en-us' might cause unintended effects.
160
